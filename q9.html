柯里-霍华德对应

目前我们看到Coq的两种用法，一种是利用归纳类型及其函数进行'编程'，另一种是用归纳命题、蕴含式、全称量词等工具'证明程序的性质'。这两种用法似乎是有着区别的，但是在Coq程序和证明中几乎是相同的。例如Inductive同时用于声明数据类型和命题，以及->同时用于描述函数类型和逻辑蕴含式。

在Coq中，可证明性表现为拥有具体的'证据'。为基本命题构造证明，实际上是以树状结构表示其证据。我们可以把证据理解为数据，命题本身理解为类型。

在 '归纳定义命题' 一节中，我们详细讨论了ev定义中存在的各种类型,我们可以从另一个角度理解这个形式化定义。

Print ev.

Inductive ev : nat -> Prop :=
    ev_0 : ev 0
    ev_SS : forall n : nat,
            ev n -> ev (S (S n)).

我们可以讲ev_0 是 ev 0 的证明，ev_SS 是 forall n :nat , ev n -> ev (S (S n))的证明。

既在类型层表达"具有...类型"，又在命题层表示"是...的证明",称为'柯里-霍华德同构(Curry-Howard correspondence)'.

柯里-霍华德同构给出了看问题的新方法，例如 ev_SS:'ev_SS 构造子接受两个参数——数字n以及命题ev n 的证明，并产生ev(S (S n))的证明';多态类型 forall X, list X 表明'可以将nil视为从某类型到该类型元素组成的空列表的函数'.

证明脚本

证明对象(proof objects)作为Coq运作的核心，当Coq执行一个证明脚本，Coq在内部逐渐构造出一个证明对象——一个类型是待证明的命题的项。Proof和Qed之间的策略告诉Coq如何构造该项。

Coq提供了Show Proof指令来显式当前证明树的状态。

Theorem ev_4'' : ev 4.
Proof.
Show Proof.
(*?Goal*)
apply ev_SS.
Show Proof.
(*(ev_SS 2 ?Goal)*)
apply ev_SS.
Show Proof.
(*(ev_SS 2 (ev_SS 0 ?Goal))*)
apply ev_0.
Show Proof.
(*(ev_SS 2 (ev_SS 0 ev_0))*)
Qed.

在任意的给定时刻，Coq都构造了一个包含?Goal的项，并且Coq知道该洞需要什么类型的证据来填补。每一个洞对应一个子目标，当没有子目标时，代表证明已经完成。证明将会被存储在全局环境中，名字为Theorem中给定的名字。

使用策略可以使证明非常方便，但是这并不是必要的。实际上，我们总能够手动构造出想要的证据,我们甚至可以通过Definition而不是Theorem来来声明证据。

Definition ev_4''' : ev 4 :=
ev_SS 2 (ev_SS 0 ev_0).

对于同一个命题来说，即使是不同的证明方式，其在全局环境中存储的证明是完全一样的。

Print ev_4.
(* ===> ev_4    =   ev_SS 2 (ev_SS 0 ev_0) : ev 4 *)
Print ev_4'.
(* ===> ev_4'   =   ev_SS 2 (ev_SS 0 ev_0) : ev 4 *)
Print ev_4''.
(* ===> ev_4''  =   ev_SS 2 (ev_SS 0 ev_0) : ev 4 *)
Print ev_4'''.
(* ===> ev_4''' =   ev_SS 2 (ev_SS 0 ev_0) : ev 4 *)

量词、蕴含式、函数

在Coq的数据结构和程序中，构造子和函数的类型中拥有箭头；而在运用证明的地方，可以使用归纳定义命题引入构造子、或者函数来为蕴含式提供证据。

Theorem ev_plus2 : 
forall n , ev n -> ev (2 + n).
Proof.
intros n H.
simpl.
apply ev_SS.
apply H.
Qed.

Print ev_plus2.
ev_plus2 = 
fun (n : nat) (H : ev n) =>
ev_SS n H : ev (2 + n)
     : forall n : nat,
       ev n -> ev (2 + n)

(*证据 = 证明对象(函数) : 命题(类型)*)

(*证明对象相当于一个函数，与下面的定义完全等价*)
Definition ev_plus2' (n : nat) (H: ev n) : ev (2 + n) :=
ev_SS n H.

Print ev_plus2'.
ev_plus2' = 
fun (n : nat) (H : ev n) =>
ev_SS n H
     : forall n : nat,
       ev n -> ev (2 + n)

可以看到，ev_plus2的证明对象是一个函数，接收两个参数，并返回一个证据。在接收的第二个参数时，其类型依赖于第一个参数n的值，这种类型称为'依赖类型'.一般地，->和forall都可以表示应用于证据的函数，当使用forall没有依赖，就可以简写为->.

以下三个定义时等价的

Definition ev_plus2'' : Prop :=
forall n , forall (E : ev n) , ev (n + 2).

Definition ev_plus2''' : Prop :=
forall n , forall (_ : ev n) , ev (n + 2).

Definition  ev_plus2'''' : Prop :=
forall n , ev n -> ev (n + 2).

提示: 注意这三个定义仅仅为了说明->与forall的使用互换，但与ev_plus2和ev_plus2'是不同的，前者是后者指向的命题（或者说前者是后者的类型），后者是前者的证据。

使用策略编程

我们已经知道既可以通过策略来完成证明，也可以手动构造完成证明;同样的，在构造程序时，我们也可以使用策略来完成之前通过手动构造的程序。

Definition add_one : nat -> nat.
intro n.
apply S.
apply n.
Defined.

Definition add_two : nat -> nat.
intro n.
apply (add_one (add_one n)).
Defined.

Compute add_one 2.
(*   = 3
     : nat*)

Compute add_two 2.
(*= 4
     : nat*)

使用策略进行定义时，用'.'进行终止，这个记号指定Coq进入证明脚本模式来构造类型时nat->nat项。通过Defined而非Qed来终止证明，这使得它可以像正常函数一样参与到计算中。

(*使用Qed的结果*)
Compute add_two 2.
(*     = add_two 2
     : nat*)

逻辑联结词作为归纳类型

在Coq中，全称量词以及作为特殊情况的蕴含式是内置的，所有其他的联结词都是被归纳定义的。

合取

Inductive and (P Q :Prop) : Prop :=
| conj : P -> Q -> and P Q.

这个定义解释了destruct 和 intros 模式能用于一个合取前提，因为只有一种构造情况conj.

类似split 策略能够用于所有只有一个构造子的归纳定义命题，特别地，能够用于and.

析取

Inductive or (P Q : Prop) : Prop :=
| or_introl : P -> or P Q
| or intror : Q -> or P Q.

析取的归纳定义有两个构造子，分别用于析取的两边。这个定义解释了destruct策略在一个析取前提上的行为。

存在量化

Inductive ex {A : Type} (P : A -> Prop ) : Prop :=
| ex_intro : forall x : A ,P x -> ex P.

构造子ex_intro提供了一个给定证据类型x和P x的证明，可以构造ex P的证据的方式。

True 和 False

Inductive True : Prop :=
| I : True.

Inductive False : Prop :=.

True拥有一个构造子，因此True的所有证据都是一样的，False没有构造子，因为没有任何方式能够构造它的证明。

相等关系

相等关系也是归纳的(标准库中给出的是轻微变体，多了些容易使用的归纳法则)。

Inductive eq {X:Type} : X -> X -> Prop :=
| eq_refl : forall x ,eq x x.

Notation "x == y" := (eq x y) (at level 70, no associativity): type_scope.

如果两个项能够通过一些简单的计算规则转换，那么Coq认为两者相等。这些计算规则，与Compute所使用的规则相似，包括函数应用的计算，定义的内联，match 语句的化简。

Lemma four : 2 + 2 = 1 + 3.
Proof.
apply eq_refl.
Qed.

实际上reflexivity策略本质相当于apply eq_refl的简写。

反演

一般地，inversion 策略：

  • 接受一个前提H，该前提的类型P是通过归纳定义的
  • 对于P的定义里每一个构造子C，
    · 产生一个新的子目标，在该子目标中假设H是通过C构造的
    · 作为额外的假设，在子目标上下文中增加C的论据（前提）
    · 将C的结论与当前的目标相匹配，计算出为了能够应用C而必须成立的一些相等关系
    · 将这些相等关系加入上下文中
    · 如果这些相等关系无法满足(如S n = 0)，那么立即解决这个子目标

例子1 : 如果我们反演一个使用or构造的前提，它有两个构 造子，所以产生了两个子目标。构造子的结论（结果类型，即P ∨ Q） 并没有对于P和Q的形式有任何要求，所以在子目标的上下文中我们不会 获得额外的相等关系。
例子2 : 如果我们反演一个使用and构造的前提，它只有一个构造子， 所以只产生一个子目标。再一次地，构造子的结论（结果类型，即P ∧ Q） 并没有对于P和Q的形式有任何要求，所以在子目标的上下文中我们不会 获得额外的相等关系。不过，这个构造子有两个额外的参数，我们能够在子 目标的上下文中看到它们。
例子3 : 如果我们反演一个使用eq构造的前提，它也只有一个构造子， 所以只产生一个子目标。但是，现在eq_refl构造子的形式给我们带来 的额外的信息：它告诉eq的两个参数必须是一样的。于是inversion策 略会将这个事实加入到上下文中。
